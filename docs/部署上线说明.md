# 部署上线说明

## 1. 本地 Docker 启动

### 1.1 启动 PostgreSQL + API

```bash
docker compose up --build db api
```

说明：

1. `api` 容器启动时会自动执行 `prisma migrate deploy`，然后再启动 NestJS 服务。
2. 默认暴露端口：`db=55432`、`api=3001`。
3. `docker-compose.yml` 默认将 `CRAWLER_ADAPTER_NAME` 设置为 `mock`，便于本地联调归档、列表、分页和任务调度链路。
4. 如需调整宿主机端口，可在启动前覆盖 `DOCKER_DB_PORT`、`DOCKER_API_PORT`。

### 1.2 启动完整服务

```bash
docker compose --profile full up --build
```

说明：

1. `full` profile 会额外启动 `web` 服务，暴露端口 `3000`。
2. `web` 容器通过 `INTERNAL_API_BASE_URL=http://api:3001` 访问 NestJS API。
3. `web` 运行时同样需要访问 PostgreSQL，以支撑 NextAuth Prisma Adapter。
4. 如需调整前端宿主机端口，可在启动前覆盖 `DOCKER_WEB_PORT`。

### 1.3 与真实 X 绑定的关系

当前容器编排主要用于本地功能联调和 MVP 验证。真实的 X 交互式绑定流程依赖宿主机可见浏览器窗口与人工登录，建议在宿主机直接运行 `api` 服务完成绑定，再继续进行真实抓取验证。

## 2. Vercel 前端部署

### 2.1 项目配置

1. 在 Vercel 中将 Frontend Project 的 Root Directory 设置为 `apps/web`。
2. 仓库内已提供 `apps/web/vercel.json`，固定使用 `Next.js` 框架和 `next build` 构建命令。
3. `src/app/api/**/*` 的 Vercel Function 超时时间已统一提升到 `30s`，便于前端 BFF 路由调用 NestJS API。
4. `apps/web/.vercelignore` 已排除测试、Playwright 结果和本地构建产物，减少部署上传体积。

### 2.2 环境变量

Vercel 中至少需要配置以下变量：

1. `NEXTAUTH_SECRET`
2. `AUTH_TRUST_HOST=true`
3. `DATABASE_URL`
4. `INTERNAL_API_BASE_URL`
5. `INTERNAL_API_SHARED_SECRET`

推荐按环境分别配置：

1. Preview：`INTERNAL_API_BASE_URL` 指向测试或预发 NestJS API。
2. Production：`INTERNAL_API_BASE_URL` 指向正式 NestJS API。
3. `DATABASE_URL` 需指向 Neon PostgreSQL，以支撑 NextAuth Prisma Adapter。

### 2.3 域名与回调地址

1. 若配置了正式自定义域名，建议显式设置 `NEXTAUTH_URL=https://你的正式域名`。
2. 若未显式设置 `NEXTAUTH_URL`，前端会自动回退使用 `VERCEL_PROJECT_PRODUCTION_URL`、`VERCEL_BRANCH_URL` 或 `VERCEL_URL` 推导站点地址，适配 Preview 与默认域名场景。
3. 若 API 有来源校验、WAF 或 IP 白名单，需要额外放通 Vercel 到 API 的服务端访问。
