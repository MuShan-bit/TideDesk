# 多平台内容抓取、AI 分类与发布平台需求文档

## 1. 文档说明

- 文档状态：更新中
- 更新时间：2026-03-20
- 项目名称：auto-x-to-wechat
- 文档目标：明确平台从“X 推荐流归档”演进到“多账号抓取、AI 分类总结、多平台发布与订阅”的产品需求、功能范围、技术边界与实施基础

## 2. 项目背景

本项目计划开发一个 Web 软件，允许用户在平台内登录并绑定自己的 x.com 账号。系统在绑定完成后，能够按设定周期抓取该账号推荐流中的帖子内容，对抓取结果进行去重与归档，并以卡片列表的形式在后台页面中进行分页展示，供用户后续检索、查看与进一步处理。

在当前 MVP 已实现“账号绑定 -> 定时抓取 -> 去重存储 -> 富文本归档 -> 列表展示”基础上，下一阶段将继续扩展为内容运营平台，新增分类与标签、AI 模型接入、自动分类打标、周报月报、多抓取模式、多 X 账号、多平台绑定、人工编辑后一键发布，以及 Telegram 订阅能力。

## 3. 产品目标

### 3.1 核心目标

1. 支持用户登录平台并绑定自己的 X 账号。
2. 支持一个用户绑定多个 X 账号，并为每个账号配置独立抓取策略。
3. 支持系统按固定周期自动抓取推荐流、热点流和自定义搜索结果。
4. 对已抓取帖子按平台帖子唯一标识去重，避免重复入库，同时保留多次命中记录。
5. 将帖子内容保存为可展示、可编辑的富文本归档内容。
6. 为归档帖子增加单分类与多标签能力，并支持人工维护。
7. 接入多种 AI 模型服务提供商，支持可配置模型与任务级模型选择。
8. 使用 AI 自动为帖子进行分类、打标签、摘要提取与阶段性汇总。
9. 支持生成周报、月报，并为后续日报扩展预留能力。
10. 支持将帖子归档或报告编辑后，一键发布到微信公众号、知乎、CSDN 等平台。
11. 支持 Telegram 订阅，将帖子或报告按规则推送给订阅目标。

### 3.2 阶段性目标（VNext）

1. 支持单个平台注册用户绑定多个 X 账号，并分别启用推荐、热点、自定义搜索抓取模式。
2. 支持维护分类体系与标签体系，帖子可被人工或 AI 赋予单个分类和多个标签。
3. 支持配置多个 AI 提供商、多个模型，并为“帖子分类”“报告生成”“发布草稿润色”等任务选择默认模型。
4. 支持对新增归档帖子执行自动分类与标签生成，并允许人工覆盖 AI 结果。
5. 支持生成周报、月报，并在页面中查看、编辑和再次触发生成。
6. 支持绑定微信公众号、知乎、CSDN 等发布渠道。
7. 支持把归档帖子或报告编辑成发布草稿并一键发布到多个目标平台。
8. 支持 Telegram 订阅与定向推送。

### 3.3 非目标

当前阶段不包含以下能力：

1. 不以支持所有内容平台为目标，优先实现 X 抓取与微信公众号、知乎、CSDN、Telegram 等重点渠道。
2. 不在当前阶段实现复杂的多成员协作、组织空间与审批流。
3. 不以“完全无人审核的全自动批量发布”作为默认交付形态，发布前保留人工编辑入口。
4. 不实现自研大模型训练，仅接入第三方模型服务。
5. 不以对外开放 API 平台为当前阶段交付目标。

## 4. 用户角色

### 4.1 普通用户

普通用户是平台的主要使用者，能够完成：

1. 登录平台。
2. 通过浏览器辅助登录绑定、查看、解绑多个 X 账号。
3. 绑定微信公众号、知乎、CSDN 等发布渠道。
4. 配置抓取周期、抓取模式和搜索条件。
5. 查看抓取记录、报告和发布记录。
6. 浏览归档帖子列表、详情、分类与标签结果。
7. 配置 AI 提供商、模型与自动化任务。
8. 编辑归档内容或报告并发布到外部平台。
9. 配置 Telegram 订阅规则。

### 4.2 系统管理员（预留）

第一阶段不强制要求实现完整后台管理端，但建议在数据模型和权限设计中预留管理员角色，以支持后续查看系统任务状态、失败日志、账号异常和全局数据统计。

## 5. 业务范围与核心流程

### 5.1 核心业务流程

1. 用户注册或登录平台。
2. 用户绑定一个或多个 X 账号，并为其配置推荐、热点或自定义搜索抓取策略。
3. 用户可选绑定微信公众号、知乎、CSDN 等外部发布渠道。
4. 系统验证绑定状态可用并自动回填账号信息。
5. 系统根据抓取配置执行定时任务，或由用户手动触发抓取。
6. 系统拉取对应模式下的帖子数据，并记录本次抓取任务及抓取到的帖子 ID。
7. 系统判断帖子唯一标识是否已归档，并记录本次命中来源。
8. 若未归档，则解析帖子内容并保存为富文本归档。
9. 系统调用 AI 模型对新归档帖子进行分类、标签、摘要等增强处理。
10. 用户在前端页面中以卡片列表分页浏览归档内容，并可按分类、标签、模式等维度筛选。
11. 系统可按周、月聚合归档数据生成 AI 报告。
12. 用户可对帖子归档或周报月报进行人工编辑。
13. 用户可一键发布到微信公众号、知乎、CSDN 等平台。
14. 系统可按订阅规则将帖子或报告发送到 Telegram。

### 5.2 去重范围

增强阶段建议按“用户 + 来源平台 + 平台帖子唯一标识”进行归档去重，并引入“命中记录”概念：

1. 同一用户下，同一个平台帖子即便来自多个 X 账号、多个抓取模式或多次抓取，也只保留一条主归档记录。
2. 每次抓取任务都需要记录抓取到该帖子的来源账号、抓取模式、搜索条件和处理结果。
3. 不同用户之间默认不共享归档主数据，避免用户间数据串扰与权限问题。

## 6. 功能需求

### 6.1 登录与权限

#### 6.1.1 登录方式

1. 平台采用 NextAuth.js 实现身份认证。
2. 第一阶段建议支持至少一种稳定登录方式，例如邮箱验证码登录或 OAuth 登录。
3. 登录后才允许访问账号绑定、抓取配置、归档内容等页面。

#### 6.1.2 权限要求

1. 用户只能访问和管理自己的绑定账号、抓取任务和归档内容。
2. 未登录用户访问受保护页面时，应跳转到登录页。
3. 后端接口必须基于用户身份进行数据隔离。

### 6.2 X 账号绑定

#### 6.2.1 绑定能力

1. 用户可在平台中发起多个 X 账号浏览器辅助绑定流程。
2. 平台能够自动拉起浏览器并跳转到 `x.com` 登录页。
3. 用户在浏览器窗口中手动完成登录后，系统自动采集账号信息与登录态 Cookie。
4. 绑定成功后，系统需展示绑定状态，例如“已绑定”“失效”“待验证”。
5. 用户可主动解绑账号。
6. 用户可在绑定失效时重新绑定。
7. 绑定列表需支持展示多个 X 账号，并区分各账号的抓取配置与最近执行状态。

#### 6.2.2 绑定信息

至少需要保存以下信息：

1. X 用户 ID。
2. X 用户名（username）。
3. 显示名称。
4. 头像地址。
5. 绑定状态。
6. 用于抓取推荐流所需的授权信息或会话凭证。
7. 最近一次验证时间。
8. 最近一次浏览器绑定会话状态与错误摘要。
9. 是否为默认抓取账号。
10. 当前启用的抓取策略集合。

#### 6.2.3 安全要求

1. 与 X 账号绑定相关的敏感凭证必须加密存储。
2. 前端不得直接暴露敏感凭证。
3. 后端日志中不得打印完整 token、cookie 或会话信息。

#### 6.2.4 待确认事项

由于“抓取推荐流”通常依赖登录态，且不一定能通过官方开放接口直接获取，第一阶段明确采用“平台拉起受控浏览器 -> 用户手动登录 -> 系统自动采集登录态 -> 保存绑定并执行抓取”的方案。后续如需远程部署，可再演进为远程浏览器或扩展采集方案。

### 6.3 定时抓取任务

#### 6.3.1 抓取配置

1. 用户可为绑定账号启用或关闭自动抓取。
2. 用户可为每个绑定账号配置一个或多个抓取策略。
3. 每个抓取策略可设置抓取周期，例如每 15 分钟、30 分钟、1 小时、2 小时。
4. 系统需记录每个抓取策略的下一次计划执行时间。
5. 系统需支持手动立即抓取一次。

#### 6.3.2 抓取目标

1. 抓取目标支持三种模式：推荐模式、热点模式、自定义搜索模式。
2. 推荐模式抓取绑定 X 账号登录后首页推荐流中的帖子内容。
3. 热点模式抓取 X 当前热点或热度较高的帖子内容，需支持区域或语言等基础参数。
4. 自定义搜索模式根据用户配置的关键词、搜索语法或主题词抓取结果，例如“科技热点”“AI agent”。
5. 当前阶段以文本内容和基础元数据抓取为主。
6. 若帖子包含图片、视频、引用帖、转发帖，系统应尽量保留结构化信息或原始引用关系。
7. 真实抓取优先基于浏览器自动化打开对应页面并抓取可见结果。

#### 6.3.3 抓取结果记录

每次抓取任务需记录：

1. 抓取任务 ID。
2. 触发方式（定时/手动）。
3. 任务开始时间与结束时间。
4. 任务状态（成功、部分成功、失败）。
5. 抓取到的帖子总数。
6. 新增归档数量。
7. 跳过数量（已存在或无效数据）。
8. 失败原因摘要。

#### 6.3.4 异常处理

1. 若账号凭证失效，应将账号状态标记为“失效”，并提示用户重新绑定。
2. 若抓取过程中出现网络异常、限流或页面结构变化，应记录失败日志。
3. 任务失败不应影响历史归档数据。
4. 对同一账号的重复执行任务需有并发控制，避免同一时刻重复抓取。

### 6.4 帖子去重与归档

#### 6.4.1 去重规则

1. 系统以“来源平台 + 平台帖子唯一标识”作为主要唯一标识。
2. 针对同一用户，若该平台帖子已存在，则本次抓取不重复归档。
3. 每次抓取仍需记录“该帖子已抓取过并被跳过”的处理结果，以及命中来源账号与抓取模式。

#### 6.4.2 归档内容范围

系统归档时，至少保存以下字段：

1. 帖子 ID。
2. 帖子链接。
3. 作者信息。
4. 原始文本内容。
5. 解析后的富文本内容。
6. 发布时间。
7. 抓取时间。
8. 媒体资源信息（如有）。
9. 帖子类型（普通帖、转发、引用帖、回复，若可识别）。
10. 原始抓取载荷快照或必要的原始 JSON 数据。
11. AI 生成的分类结果。
12. AI 生成的标签结果。

#### 6.4.3 富文本归档要求

1. 系统需要将帖子内容转换为适合前端展示的富文本格式。
2. 富文本需尽量保留以下内容：段落与换行、链接、提及账号、话题标签、图片或视频的引用位置、引用帖或转发帖的关系说明。
3. 建议同时保存一份结构化 JSON 和一份渲染后的 HTML 或可序列化富文本数据，以兼顾展示与后续扩展。

#### 6.4.4 数据一致性

1. 帖子归档写入与抓取记录写入需在逻辑上保持一致。
2. 归档成功后再将帖子标记为已处理。
3. 若归档过程中失败，应保留失败状态，允许后续补偿处理。

### 6.5 归档内容展示

#### 6.5.1 列表页

系统需提供帖子归档列表页，采用卡片列表展示，支持：

1. 分页浏览。
2. 每条卡片展示作者、发布时间、摘要内容、媒体缩略图和来源链接。
3. 展示归档时间。
4. 展示帖子类型标签。
5. 点击进入详情页。

#### 6.5.2 详情页

帖子详情页至少展示：

1. 完整富文本内容。
2. 作者信息。
3. 原始发布时间。
4. 归档时间。
5. 媒体资源。
6. 原始链接。
7. 所属抓取任务信息。

#### 6.5.3 列表筛选（建议）

建议支持以下筛选条件：

1. 按绑定账号筛选。
2. 按抓取时间范围筛选。
3. 按帖子类型筛选。
4. 按关键词搜索。
5. 按抓取模式筛选。
6. 按分类筛选。
7. 按标签筛选。

### 6.6 分类与多标签

#### 6.6.1 归档分类

1. 每条归档帖子支持 0 或 1 个主分类。
2. 分类支持系统内置分类与用户自定义分类。
3. 用户可手动修改分类，人工结果应覆盖 AI 推荐结果。

#### 6.6.2 标签体系

1. 每条归档帖子支持 0 到多个标签。
2. 标签支持用户自定义创建、合并、停用与重命名。
3. 标签可来源于人工添加、AI 生成或规则映射。

#### 6.6.3 列表与详情展示

1. 归档列表卡片需展示分类与标签摘要。
2. 归档详情页需展示完整分类、标签、AI 推荐来源和最近更新时间。
3. 分类与标签应支持作为筛选条件使用。

### 6.7 AI 模型服务接入

#### 6.7.1 提供商支持

1. 系统需支持多种 AI 模型服务提供商。
2. 优先支持官方提供商与 OpenAI 兼容接口两类接入方式。
3. 提供商配置至少包括：提供商类型、名称、Base URL、API Key、启用状态、备注。

#### 6.7.2 模型配置

1. 每个提供商下可配置一个或多个模型。
2. 模型可按任务维度指定默认值，例如帖子分类模型、报告生成模型、发布润色模型。
3. 系统需支持启用、停用、连通性测试和失败回退策略。

#### 6.7.3 安全要求

1. API Key、Secret 等敏感凭证必须加密存储。
2. 前端不得直接展示完整密钥内容。
3. 所有 AI 调用记录需保留基础审计信息，但不记录完整敏感凭证。

### 6.8 AI 自动分类与标签

#### 6.8.1 自动化触发

1. 新帖子归档成功后，系统可自动触发 AI 分类与标签任务。
2. 用户可手动对单条帖子重新执行 AI 分类。
3. 系统需支持批量重跑，以便模型升级后重新打标。

#### 6.8.2 输出结果

1. AI 输出至少包括：主分类、标签列表、简要摘要、置信度或可解释说明。
2. 系统需保存最近一次 AI 任务状态、输入模型、输出结果和错误信息。
3. 用户人工修改后，系统需标记该结果为“人工覆盖”。

### 6.9 报告模块

#### 6.9.1 报告类型

1. 系统至少支持周报、月报。
2. 报告体系需为后续日报扩展预留能力。
3. 用户可按绑定账号、抓取模式、分类、标签、时间区间生成报告。

#### 6.9.2 报告内容

1. 报告需聚合周期内的帖子数据统计。
2. 报告需支持 AI 自动总结，包括热点主题、重点观点、代表帖子与可发布选题建议。
3. 报告内容应以富文本方式存档，并支持后续编辑。

#### 6.9.3 报告管理

1. 系统需提供报告列表页和详情页。
2. 用户可重新生成报告、编辑报告、导出报告或发起发布。

### 6.10 拉取模式

#### 6.10.1 推荐模式

1. 推荐模式抓取绑定 X 账号在登录状态下的首页推荐内容。
2. 推荐模式需保留当前已实现的抓取能力。

#### 6.10.2 热点模式

1. 热点模式抓取 X 热度较高或趋势帖文。
2. 热点模式建议支持语言、地区或主题范围等基础参数。

#### 6.10.3 自定义搜索模式

1. 用户可配置一个或多个搜索条件。
2. 搜索条件支持关键字、短语或 X 搜索语法组合。
3. 每个搜索条件可独立配置抓取周期和启停状态。

### 6.11 多平台绑定

#### 6.11.1 绑定范围

1. 平台需支持绑定多个 X 账号作为抓取源。
2. 平台需支持绑定微信公众号、知乎、CSDN 等作为发布目标渠道。
3. 后续其他平台可通过扩展适配器方式接入。

#### 6.11.2 渠道能力

1. 每个绑定渠道需声明能力范围，例如 `SOURCE`、`PUBLISH`、`SUBSCRIBE`。
2. 渠道绑定需支持重新授权、禁用、解绑和最近一次校验状态查看。

### 6.12 内容编辑与一键发布

#### 6.12.1 发布内容来源

1. 用户可从归档帖子生成发布草稿。
2. 用户可从周报、月报生成发布草稿。
3. 系统应支持把多条帖子组合成一篇待发布内容。

#### 6.12.2 编辑能力

1. 发布前必须支持人工编辑标题、摘要、正文、标签和封面。
2. 系统应保留平台源帖链接与引用关系，避免内容来源丢失。
3. 可选支持 AI 润色，但不应替代人工最终确认。

#### 6.12.3 发布执行

1. 用户可选择一个或多个目标平台一键发布。
2. 系统需记录每次发布结果、目标平台返回的文章 ID / URL、错误摘要和发布时间。
3. 发布失败时支持重试。

### 6.13 多 X 账号

1. 单个平台注册用户可绑定多个 X 账号。
2. 每个 X 账号可独立配置抓取策略、启停状态和默认用途。
3. 归档列表与报告生成时支持跨账号聚合，也支持按单账号筛选。

### 6.14 Telegram 订阅

1. 系统需支持 Telegram 订阅能力。
2. 订阅对象可选择接收新增归档、周报、月报或指定分类 / 标签内容。
3. 订阅规则应支持频率配置、目标 Chat / Channel 配置和启停控制。
4. 系统需记录投递结果与失败原因。

### 6.15 页面体验与国际化

#### 6.15.1 页面内容表达

1. 首页及关键页面文案应面向最终用户，优先介绍产品功能、使用价值与系统能力。
2. 页面中不应出现“当前阶段”“MVP 准备中”“推进路径”等偏研发过程的说明性文案。
3. 首页应突出真实 X 绑定、自动抓取、AI 分类、报告生成、跨平台发布等核心能力。
4. 首页可适度展示技术栈与部署方式，用于增强产品可信度和可维护性表达。

#### 6.15.2 Header 与导航体验

1. Header 需拆分为品牌区、功能导航区、工具区，避免导航项与切换控件拥挤在同一行。
2. 主导航需始终显示当前页面的选中状态，例如保留底色、高亮文字或其它持续可见的视觉反馈。
3. 语言切换与主题切换应归入工具区，不应影响主导航的可读性与点击区域。

#### 6.15.3 中英文国际化

1. 系统支持简体中文与英文两套界面语言。
2. 用户点击语言切换后，应在当前页面即时生效。
3. 语言偏好需在刷新页面、重新访问系统后继续保留。
4. 页面标题、按钮、提示语、状态文案、分页信息和空状态文案都需随语言切换同步更新。

#### 6.15.4 亮色与暗色主题

1. 系统支持亮色主题与暗色主题切换。
2. 主题切换应即时生效，并在后续访问中保留用户选择。
3. Header、页面背景、卡片、空状态、错误状态、导航项和主要按钮需适配双主题。

### 6.16 抓取记录展示

系统需提供抓取任务记录页面，至少包括：

1. 任务执行时间。
2. 触发方式。
3. 任务状态。
4. 抓取总数。
5. 新增数。
6. 跳过数。
7. 错误摘要。
8. 抓取模式。
9. 来源账号。

用户可点击单次任务查看更详细的处理结果。

## 9. 接口与任务设计建议

### 9.1 前端应用

前端采用以下技术：

1. Next.js
2. TypeScript
3. Tailwind CSS
4. ShadCN UI
5. NextAuth.js

前端主要负责：

1. 登录与会话管理。
2. 账号绑定与多平台渠道管理页面交互。
3. 抓取任务配置与触发。
4. 归档内容列表、详情、分类与标签维护。
5. AI 模型配置、报告管理、发布草稿编辑与 Telegram 订阅配置。

### 9.2 后端服务

后端采用以下技术：

1. NestJS
2. Prisma
3. PostgreSQL

后端主要负责：

1. 用户业务数据管理。
2. X 抓取源账号与多平台发布渠道绑定信息管理。
3. 抓取任务调度与执行。
4. 帖子解析、去重、分类、标签与归档。
5. AI 调用编排、报告生成、发布执行与订阅投递。
6. 为前端提供 REST API 或 GraphQL API。

### 9.3 定时任务建议

1. 建议由 NestJS 定时任务模块或队列系统执行抓取任务。
2. 若后期抓取量增大，建议引入队列，例如 BullMQ。
3. 调度器应支持失败重试、超时控制和并发限制。
4. AI 分类、报告生成、发布执行和 Telegram 投递建议与抓取任务统一接入异步任务体系。

## 10. 非功能需求

### 10.1 安全性

1. 敏感凭证加密存储。
2. 服务端接口必须进行身份校验和权限校验。
3. 防止越权访问其他用户数据。
4. 支持基础操作审计日志。
5. AI 模型密钥、第三方平台发布密钥和 Telegram 订阅密钥都应使用统一密钥管理策略。

### 10.2 性能

1. 归档列表页在常规分页场景下应快速返回。
2. 归档表需建立合理索引，至少包括 userId、sourcePlatform、sourcePostId、archivedAt。
3. 抓取任务不应阻塞用户正常访问。
4. AI 任务、报告任务和发布任务需要具备排队、限流和成本控制能力。

### 10.3 可维护性

1. 前后端项目结构需清晰分层。
2. Prisma 数据模型需具备明确迁移方案。
3. 关键模块需补充基础测试。
4. 新增平台与新增模型应尽可能通过适配器扩展，而不是侵入式修改核心流程。

### 10.4 可观测性

1. 记录抓取任务运行日志。
2. 记录失败原因和异常栈。
3. 为后续监控告警预留接口。
4. 记录 AI 调用、报告生成、发布执行与 Telegram 投递的关键日志与状态。

## 11. 部署要求

### 11.1 开发与版本管理

1. 代码使用 Git 进行版本管理。
2. 建议建立基础分支策略和提交规范。

### 11.2 部署方案

支持以下部署方式：

1. Docker 部署完整服务。
2. 前端部署至 Vercel。
3. PostgreSQL 使用 Neon 托管。

### 11.3 部署建议

1. 若抓取逻辑依赖浏览器自动化或长时间任务，需要确认 Vercel Serverless 是否满足执行需求。
2. 若不满足，建议将抓取服务独立为容器化后台服务，前端仍部署在 Vercel。
3. 数据库统一使用 PostgreSQL，保证前后端环境一致。
4. AI 网关、发布 Worker、Telegram 投递 Worker 建议与抓取 Worker 统一部署在 Docker 后台服务中。

## 12. 验收标准

### 12.1 当前 MVP 验收标准

当以下条件满足时，可认为第一阶段 MVP 基本完成：

1. 用户能够成功登录平台。
2. 用户能够绑定一个可用的 X 账号。
3. 系统能够按设定周期执行抓取任务。
4. 系统能够记录每次抓取结果。
5. 同一绑定账号下的帖子不会因重复抓取而重复入库。
6. 新抓取到的帖子会被解析并归档为可展示内容。
7. 用户能够在页面中以卡片列表查看归档帖子并进行分页浏览。
8. 用户能够查看某条帖子的详情内容。

### 12.2 增强阶段验收标准

当以下条件满足时，可认为增强阶段主要能力可进入联调与灰度：

1. 用户能够绑定多个 X 账号，并独立配置推荐、热点、自定义搜索模式。
2. 系统能够对归档帖子维护单分类与多标签，并支持人工修改。
3. 用户能够配置多个 AI 提供商和模型，并为不同任务设置默认模型。
4. 新归档帖子能够自动触发 AI 分类与标签生成。
5. 系统能够生成周报、月报，并支持重新生成与人工编辑。
6. 用户能够绑定微信公众号、知乎、CSDN 等至少一个外部发布渠道。
7. 用户能够把归档帖子或报告编辑后发布到目标平台，并看到发布结果。
8. 用户能够配置 Telegram 订阅并收到至少一种类型的内容推送。

## 13. 风险与待确认项

### 13.1 高优先级风险

1. X 推荐流抓取能力可能受登录方式、页面结构变化、限流与风控影响。
2. 若采用浏览器自动化，部署资源和稳定性成本会提高。
3. 若采用会话凭证存储方式，需要重点评估安全和合规边界。
4. 第三方 AI 提供商接口稳定性、计费策略和响应质量会直接影响自动分类和报告结果。
5. 外部平台发布接口可能受风控、页面结构变化或账号权限限制影响。

### 13.2 待确认问题

1. 热点模式的“热度来源”与区域定义使用哪种 X 页面或接口视图。
2. 自定义搜索模式是否允许使用高级搜索语法以及是否支持多语言查询。
3. 是否需要同步保存图片、视频到自有存储，还是仅保存外链。
4. 富文本格式最终采用 HTML、Slate JSON、TipTap JSON 还是自定义结构。
5. AI 分类结果与人工分类冲突时，是否需要保留版本历史与回滚能力。
6. 微信公众号、知乎、CSDN 的发布能力优先级与接入顺序如何确定。
7. Telegram 订阅是走 Bot 私聊、频道推送，还是二者都支持。
